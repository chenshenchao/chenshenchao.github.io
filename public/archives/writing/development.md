## 错误累积、延后大爆发与数据治理

程序开发中很多的错误并非是单一的原因造成的，往往是由不同的漏洞引入了错误的数据，错误的数据在整个系统中流转，到达某个对数据准确度有强要求的程序时爆发。
大部分系统都有一定的容错机制，而这种容错机制就是错误积累的根源，因为错误被引入而不纠错，多个错误叠加后就集中爆发了。
所以程序系统是需要有数据治理机制的，需要有程序不断地检查数据的准确性，同时又有程序不断地对数据进行纠错。

## 旧观念与刻板印象

### 脚本语言处理数据

刻板印象里，用脚本语言处理数据是一种很常见的做法。
但是脚本语言一般是弱类型，在处理简单数据结构是很方便，但是一旦处理复杂数据结构时，缺少强类型系统的复制，反而效率是下降的。
当配合的数据是从一个强类型系统，比如 MYSQL 这种关系型数据库或者明确接口类型时比较合适，因为可以以强类型一方的类型为依据认准一方就可以了。
而当配合的同样是一个弱类型系统，此时，类型几乎处于丢失状态，双方对类型都没有太多的检查，这个时候就会没有依据地频繁出现类型对不上，数据错乱等问题。

所以刻板印象是把同步数据，数据处理和脚本语言挂钩，而其本质是：
1. 脚本语言适合处理简单结构，不适合处理复杂结构。
2. 脚本语言适合和强类型配合，不适合和弱类型配合。

### 前端比后端简单

这也是十几年前的刻板印象，其实各端的难度是要看业务场景的。
后端大部分时间都是增删改查，服务器环境稳定。
前端页面组件多变，浏览器环境版本多不稳定，兼容问题频出。
对于大多数的小公司而言，其难点都在前端。
高并发、分布式等后端问题一般要到了公司有一定规模后才会出现，小公司的服务卡，很大程度是服务器配置太低。
比如使用 PHP 作为开发语言的项目，性能本就不是其考虑的重点（不然不会选PHP，PHP性能优化的核心在于让出任务，比如写长SQL把计算推给MySQL，把缓存堆到 Redis 减少自身内存占用，其本身只写胶水代码，PHP 的运算速度远低于 go 、c/c++ 这类语言，能用 PHP 的 C 扩展完成计算效率会高出纯 PHP 写的算法很多）。

所以刻板印象是前端简单，后端难，实则后端难的是大公司的需求，而前端从小公司需求难度就很高了。
1. 张冠李戴的难度匹配，把后端难的哪些大公司业务场景，当做所有后端都难，实则很多小公司就不用这些。
2. 时间跨度的刻板印象，前端简单的印象是十几年前留下的，那个时候 web 前端确实难度低，但是十几年的前端卷，导致前端难度上升了好几倍。
